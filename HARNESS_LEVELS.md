# 하니스 레벨에 대한 고찰 — AI를 감싸는 구조물은 어디까지 쌓아야 하는가

> SCPC 2026 회고([본편](RETROSPECTIVE.md))의 스핀오프.
> 대회 내내 나는 하니스를 두 개 만들고 있었다. 하나는 대회가 요구한 하니스(task를 읽고 판단하는 파이썬 프로그램), 다른 하나는 **AI를 감싸는 하니스**(스킬, 래칫, 시도 레지스트리, 로그, 실험 규율). 이 글은 두 번째 하니스에 대한 이야기다. 결론부터 말하면 — **하니스와 성능의 관계는 단조증가가 아니라 역U 곡선이고, 나는 대회 후반에 곡선의 오른쪽 내리막을 실측했다.**

---

## 1. 하니스 레벨의 정의

AI와 일해본 사람이라면 누구나 어떤 형태로든 하니스를 만든다. 명시적이든 아니든. 나는 이번 대회에서 쌓아올린 순서 그대로 레벨을 다섯 단계로 정의할 수 있다고 본다.

**L0 — 맨몸 (ad-hoc 프롬프팅).**
그냥 시킨다. "이 대회 문제 풀어줘." 세션이 끝나면 아무것도 남지 않는다. 같은 실수를 무한 반복하고, 어제의 발견을 오늘 잊는다.

**L1 — 기준 문서.**
목표, 규칙, 채점 구조, 제약을 문서로 박는다(CLAUDE.md, OVERVIEW.md). AI의 모든 판단이 참조할 수 있는 불변의 좌표계. 여기서부터 "대화"가 "프로젝트"가 된다.

**L2 — 루프.**
측정→진단→개선→평가→기록의 사이클을 스킬로 만든다. AI가 "다음에 뭘 하지?"를 매번 새로 고민하지 않고, 검증된 절차를 반복한다. 진행이 리듬을 얻는다.

**L3 — 자기구속 장치.**
여기가 질적 도약이다. AI가 **자기 자신을 믿지 못하게 만드는** 장치들:
- 래칫: 교차검증 최고 수위를 넘는 변경만 살아남는다. "좋아진 것 같은데요"의 사망.
- 시도 레지스트리: 기각된 가설은 지문과 함께 봉인, 재시도 차단.
- 단일변수 실험: 제출 1회 = 가설 1개. 귀속 불가능한 변경 금지.

**L4 — 운영 시스템.**
실험 토글 체계, 변형 카탈로그, run sheet, 결정 트리, 경로 텔레메트리, 번들 대수. 이쯤 되면 프로젝트가 하나의 작은 연구소다. 모든 것이 추적되고, 모든 것이 절차화된다.

나는 L4까지 갔다. 그리고 L4에서 이상한 것을 목격했다.

## 2. 각 레벨에서 실제로 일어난 일

숫자로 보자. 리더보드 궤적에 레벨을 겹치면:

```
L0~L1 구간:  0.0882 → 0.3295   (기준 문서와 첫 구현)
L2 구간:     0.3295 → 0.583    (루프가 돌기 시작, 생성기 해독 가속)
L3 구간:     0.583  → 0.8496   (래칫+단일변수 실험의 전성기)
L4 구간:     0.8496 → 0.8496   (15회 중 마지막 7회 제출, 신기록 +0.0009 한 번)
```

물론 이건 불공정한 비교다. 뒤로 갈수록 문제가 어려워지니까(쉬운 점수는 먼저 먹힌다). L4 구간의 정체를 하니스 탓으로만 돌릴 수는 없다. 그걸 인정하고 시작한다.

하지만 하니스 고도화가 **모델의 사고 품질 자체를 갉아먹는 메커니즘**이 분명히 존재했고, 나는 그걸 세 가지로 특정할 수 있다.

---

## 3. 내리막의 세 가지 메커니즘

### 3-1. 결론의 화석화 (기록 시스템의 배신)

시도 레지스트리는 L3의 핵심 장치였고, 실제로 AI의 최대 약점인 "같은 아이디어 재탕"을 완벽히 막았다. 그런데 예상 못 한 부작용이 있었다.

레지스트리에 적힌 "이 축은 신호부재"라는 기각 기록은, 정확히는 **"내가 시도한 분석 층위에서는 신호부재"**라는 뜻이다. 그런데 기록에는 층위 정보가 없다. 시간이 지나면 그 기록은 "저긴 파봤자 없음"이라는 검증된 사실처럼 읽힌다. 모델은 세션을 시작할 때마다 이 목록을 읽고, 목록이 길수록 탐색 공간을 스스로 좁힌다.

이 대회 최대의 점수 점프 두 번(합쳐서 리더보드 +0.05 이상)은 **둘 다 레지스트리에 "신호부재"로 봉인된 영역을 사람이 강제로 다시 열어서** 나왔다. record "타입" 수준에서 없다던 신호가 "값 조합" 수준에서 완전 판별자로 나타났고, 한 번 더 내려간 "형태소" 수준에서 또 나왔다.

교훈: **기록 시스템은 사실만 보존하지 않는다. 결론도 보존한다. 그리고 사실과 달리 결론에는 유통기한과 적용 층위가 있다.** 이걸 태깅하지 않은 기록은 시간이 지나면 지식이 아니라 벽이 된다.

### 3-2. 자기 시뮬레이션 비용 (복잡도가 지능을 잠식한다)

대회 초반, 하니스 코드가 규칙 열 개짜리일 때 Fable의 영향 추정은 꽤 정확했다. "이 규칙을 넣으면 dev에서 이 task들이 바뀔 것" — 맞았다.

후반, 규칙 30개 + 우선순위 폭포(cascade) 구조가 되자 추정이 체계적으로 무너졌다. 새 규칙의 도달 범위를 "45건"이라 예측하면 실측은 6건이었다(상류 규칙들이 선점). "65건" 예측도 실측 6건. 오차가 아니라 **10배짜리 착시**다. 원인은 명확하다. 모델은 새 규칙을 고립된 명제로 추론하지, 30개 규칙이 순서대로 선점하는 시스템의 일부로 시뮬레이션하지 못한다.

이건 모델 규모의 문제가 아니라고 생각한다. 시스템 복잡도가 어느 선을 넘으면 그 시스템의 정확한 머릿속 실행은 누구에게도(사람 포함) 불가능하고, 유일한 해법은 **실측**이다. 우리도 결국 그렇게 갔다 — 모든 추정을 "일단 돌려서 diff 개수를 세는" 것으로 대체했다. 하지만 이 전환이 자동으로 일어나지 않았다는 게 문제다. 모델은 복잡해진 시스템 앞에서도 계속 자신 있게 머리로 추정했고, 실측 강제는 사람이 걸어야 했다.

여기에 운영 부채가 얹힌다. 토글이 열 개를 넘어가자 토글끼리 충돌하는 버그(`"E5X"`가 `"E5"` 부분문자열 검사에 걸림)를 모델이 직접 심었고, 구버전 코드로 만든 실험 파일을 낼 뻔했고, 검증 스크립트의 버그가 실험 결과 판독을 두 번 오염시킬 뻔했다. **하니스가 커질수록 모델의 실수는 문제 영역에서 인프라 영역으로 이동한다.** 인프라 실수가 더 위험한 이유는, 문제 실수는 점수가 알려주지만 인프라 실수는 잘못된 결론을 조용히 만들기 때문이다.

### 3-3. 문서 중력 (집행자화)

L4쯤 되면 세션을 시작할 때 모델이 읽어야 할 것이 많다. 수백 줄의 LOG, 백로그, run sheet, 결정 트리. 이것들은 연속성을 보장하는 대신 **컨텍스트와 주의를 선점한다.**

정성적인 관찰이지만 뚜렷했다: 후반의 Fable은 초반보다 눈에 띄게 "계획의 집행자"처럼 행동했다. 백로그에 적힌 다음 항목을 수행하고, run sheet의 분기를 따라가고, 로그 형식에 맞춰 기록했다. 다 올바른 행동인데, **"백로그에 없는 생각"의 빈도가 줄었다.** 초반의 그 왕성한 "이것도 봐볼까요, 저것도 이상한데요"가 사라졌다. 하니스가 요구하는 인지 노동이 탐색에 쓰일 예산을 먹은 것이다.

아이러니한 건, 후반의 돌파구들이 전부 하니스 바깥에서 왔다는 점이다. "카테고리 하나를 통으로 날리는 거 아냐?"(사람의 직감), "다시 파봐"(동결 무시 지시), "이제 목표는 top 40이야"(목표 함수 교체). 하니스는 이 중 무엇도 생성하지 못했다. 정의상 못 한다 — 하니스는 과거의 결론으로 만든 구조물이니까.

## 4. 그럼에도 하니스가 없었다면

공정하게 반대편도 쓰자. 이 대회에서 하니스 없이는 0.84는커녕 0.6도 못 갔다고 확신한다.

- 래칫이 없었으면 dev 과적합 변경들이 수십 번 리더보드를 오염시켰을 거다.
- 시도 레지스트리가 없었으면 같은 실패를 서너 번씩 반복했을 거다. (실제로 레지스트리 도입 전, 초기에 그 조짐이 있었다.)
- 단일변수 실험 규율이 없었으면 15번의 제출은 "뭐가 좋았는지 모르는" 노이즈 더미가 됐을 거다. 특히 마지막의 번들 대수(겹치는 번들 세 개의 점수 차로 성분 효과를 연립방정식처럼 역산하는 것)는 정밀한 기록 없이는 불가능했다.
- 첫 리더보드 붕괴(dev 0.65 / LB 0.34)에서 배운 "발화율 확인" 같은 교훈이 프로세스로 박제되지 않았으면, 같은 사고가 재발했을 거다.

그러니까 문제는 "하니스냐 아니냐"가 아니다. **하니스의 어떤 성분이 성능을 만들고 어떤 성분이 성능을 깎느냐**다. 내 결산은 이렇다:

| 성분 | 효과 | 이유 |
|---|---|---|
| 기준 문서 (L1) | 순증 | 좌표계는 아무리 커도 해가 없다 |
| 루프 (L2) | 순증 | 리듬은 탐색을 대체하지 않는다 |
| 래칫, 단일변수 실험 (L3) | **큰 순증** | 모델의 약점만 정확히 막는다 |
| 시도 레지스트리 (L3) | 순증이되 **독이 섞임** | 반복은 막지만 결론을 화석화한다 |
| 토글/변형/절차 체계 (L4) | 중립~순감 | 운영 부채 + 자기 시뮬레이션 실패 유발 |
| 비대해진 로그/백로그 (L4) | 순감 | 문서 중력, 집행자화 |

## 5. 설계 원칙 — 다음에 하니스를 만든다면

이번 경험에서 뽑아낸, 다음 프로젝트에 그대로 적용할 원칙들.

**1. 결론에 층위와 유통기한을 태깅하라.**
"신호부재"라고 쓰지 마라. "record 타입 수준에서 신호부재 (값 수준, 형태소 수준 미시도)"라고 써라. 기각 기록에는 반드시 "어떤 층위에서, 어떤 방법으로" 기각됐는지를 남기고, 주기적으로 미시도 층위가 남은 항목을 재개봉하는 절차를 하니스 자체에 넣어라. 재시도 금지가 아니라 **재공략 조건 명시**가 맞는 설계다.

**2. 시스템이 커지면 추정을 금지하고 실측을 강제하라.**
규칙이 N개를 넘는 순간부터 "이 변경의 영향 범위" 추정은 금지. 무조건 돌려서 diff를 센다. 이걸 모델의 재량에 맡기지 말고 하니스의 체크리스트에 박아라. 모델은 복잡한 시스템 앞에서도 자신감을 잃지 않는다 — 그게 문제다.

**3. 하니스 다이어트를 주기에 넣어라.**
로그는 자라기만 한다. N회 반복마다 "지금도 유효한 결론만 남기고 압축"하는 단계를 하니스에 내장해라. 특히 세션 시작 시 모델이 읽는 문서의 총량에 예산을 정해라. 컨텍스트는 탐색의 원료다.

**4. 인간은 하니스 바깥에 서라.**
하니스는 과거의 결론으로 만든 구조물이라, 구조물 자체를 의심하는 기능은 가질 수 없다. "지금 파는 층이 틀렸다", "목표가 바뀌었다", "그 결론 수상하다" — 이 메타 판단의 자리를 하니스 안으로 옮기려 하지 마라. 옮기는 순간 그것도 절차가 되고, 절차가 된 의심은 의심이 아니다. 이번 대회에서 가치를 만든 내 개입은 전부 하니스가 규정하지 않은 발화였다.

**5. 하니스 레벨을 문제의 단계에 맞춰라.**
이게 종합 결론이다. 탐색 단계(문제 구조를 모름)에는 낮은 레벨(L1~L2)이 맞다 — 모델의 신선한 탐색이 최대 자산이니까. 수확 단계(구조를 알고 개선을 쌓는 중)에는 L3가 정확히 맞다 — 이때 모델의 낙관과 반복이 최대 리스크니까. L4는 마감 직전, 자원(제출 횟수)이 극도로 희소할 때만 잠깐 세우는 가설 검증 인프라로 쓰고, 끝나면 해체해라. **상시 L4는 연구소가 아니라 관료제다.**

## 6. 맺음 — 두 하니스는 같은 병에 걸린다

마지막으로, 처음에 말한 "하니스가 두 개"라는 이야기로 돌아가자.

대회용 하니스(파이썬 프로그램)의 한계는 dev 120문제가 주는 정보의 상한이었다. 본 적 없는 조합 앞에서, 아무리 정교한 규칙 폭포도 결국 과거 데이터의 화석이었다.

AI용 하니스(프로세스)의 한계도 정확히 같았다. 과거 iteration들이 주는 결론의 상한. 본 적 없는 접근이 필요한 순간, 아무리 정교한 레지스트리와 run sheet도 결국 과거 결론의 화석이었다.

**하니스란 과거를 미래에 강제하는 장치다.** 그래서 과거가 미래와 닮은 구간에서는 강력하고, 닮지 않은 구간에서는 정확히 그만큼 해롭다. 좋은 하니스 설계란 결국 "우리의 과거가 어디까지 유효한가"를 아는 문제고, 그건 하니스가 스스로 답할 수 없는 질문이다.

그 질문을 계속 던지는 것. 이번 대회에서 확인한, 아직은 사람의 일이다.
