# 하니스 레벨에 대한 고찰 — AI를 감싸는 구조물은 어디까지 쌓아야 하는가

> SCPC 2026 회고([본편](RETROSPECTIVE.md))의 스핀오프.
> 대회 내내 나는 하니스를 두 개 만들고 있었다. 하나는 대회가 요구한 하니스(task를 읽고 판단하는 파이썬 프로그램), 다른 하나는 **AI를 감싸는 하니스**(스킬, 래칫, 시도 레지스트리, 로그, 실험 규율). 이 글은 두 번째 하니스에 대한 해부다.
> 결론부터: **하니스와 성능의 관계는 단조증가가 아니라 역U 곡선이다.** 나는 이번 대회에서 곡선의 오르막과 내리막을 둘 다 실측했고, 오류 사례 일곱 개의 부검 기록과 함께, 다음에 더 쌓으려면 어디를 쌓고 더 깎으려면 어디를 깎을지까지 정리한다.

---

## 1. 내 하니스의 실제 구조 — 5계층 해부

먼저 "하니스"라고 뭉뚱그려 부르던 것의 실체부터. 내 환경에서 AI(Claude Code, Fable 5)가 세션을 시작하면 실제로 이런 계층이 순서대로 작동한다.

```
[Layer 0] 전역 설정  ~/.claude/rules/common/
          ├ coding-style.md   (불변성, 파일 크기, 에러 처리 원칙)
          ├ testing.md        (커버리지 80%, 테스트 3종 요구)
          ├ performance.md    (모델 선택 가이드, 컨텍스트 관리)
          └ hooks.md          (git 커밋 컨벤션, PR 워크플로)
          → 모든 프로젝트에 무조건 주입되는 "헌법"

[Layer 1] 프로젝트 기준 문서
          ├ CLAUDE.md         (대회 구조, 채점 모델, 실격 규칙, 명령어)
          └ OVERVIEW.md       (규칙·목표 한국어 정리)
          → 이 프로젝트의 "좌표계". 세션마다 자동 로드

[Layer 2] 루프 스킬  .claude/skills/scpc-harness-tuner/
          ├ workflow.md + step-00~06 (부트스트랩→측정→진단→개선→평가→기록→통합)
          └ templates/
          → "어떻게 일할 것인가"의 절차

[Layer 3] 상태 파일  improvement/
          ├ ratchet.json      (CV/LB 최고 수위 — 이걸 넘어야만 KEEP)
          ├ attempts.jsonl    (기각된 가설의 지문 + 사유 — 재시도 차단)
          ├ backlog.md        (다음 할 일, 우선순위)
          ├ LOG.md            (iteration별 가설→측정→판정 전량)
          ├ folds.json        (5-fold 세션 CV 분할 고정)
          └ snapshots/        (KEEP 시점 코드 박제)
          → 세션을 넘어 살아남는 "기억"

[Layer 4] 실험 인프라
          ├ SCPC_EXP 토글 시스템 (코드 내 실험 분기)
          ├ experiments/ 변형 카탈로그 + run sheet (제출 결정 트리)
          └ tests/ 회귀 잠금 4종
          → 리더보드 제출을 실험으로 만드는 장치
```

레벨(L0~L4)과 계층(Layer 0~4)이 대충 맞아떨어지는 건 우연이 아니다. 프로젝트가 진행되며 아래 계층부터 순서대로 쌓았기 때문이다. 그리고 성능 기여도는 계층 순서와 무관했다 — 이게 이 글의 핵심 주장이고, 근거는 아래 부검 기록들이다.

## 2. 부검 기록 — 오류, 신호, 수정, 그리고 내가 가져갈 것

일곱 건. 각각 **무슨 오류가 났고 / 어떤 신호로 발견했고 / 뭘 고쳤고 / 일반화하면 뭐가 되는지**의 형식으로 쓴다.

### 사례 1. dev 문구 암기 → 리더보드 붕괴

- **오류**: 프롬프트의 "단, ~하라" 절 해석 규칙을 dev 데이터의 문구 그대로 짰다. dev 점수 0.650.
- **신호**: 리더보드 0.3442. **로컬-실전 괴리 0.3.** 이 신호의 특징은 하니스 내부에선 절대 안 보인다는 것 — 래칫도 CV도 dev 안에서만 돌기 때문에 dev 과적합을 원리적으로 못 잡는다. 외부 채점(제출)만이 이걸 드러냈다.
- **수정**: ① 규칙을 문구가 아닌 의미 계열(concept family)로 재작성 ② **"모든 신규 규칙은 배포 전 실전 데이터에서 발화 횟수를 센다"를 스킬 절차(step-04)에 명문화.** 정답을 보는 게 아니라 파서가 입력에 반응하는지만 세는 것이라 규정 위반도 아니다.
- **가져갈 것**: 하니스에는 반드시 **외부 접점 게이트**가 있어야 한다. 내부 지표(dev, CV)가 아무리 정교해도 그건 닫힌 계다. 닫힌 계의 지표가 오를수록 확신은 커지는데, 그 확신의 방향이 맞는지는 계 바깥만 안다.

### 사례 2. 서수 결합 버그 — 래칫이 잡고, 테스트가 박제한다

- **오류**: 이력에서 "두 번째 후보만 확정했다"의 서수로 focal을 고르는 규칙을 넓혔더니, "**첫 번째**와 세 번째 후보는 보류했고, 두 번째 후보만 확정했다"라는 문장에서 앞의 "첫 번째"에 먼저 매칭했다.
- **신호**: dev 0.737 → 0.687. **래칫이 즉시 걸렸다.** 최고 수위 미달 = 자동 기각. 이 신호의 특징은 빠르고 기계적이라는 것 — 사람이 diff를 들여다보기 전에 시스템이 "이건 아니다"를 선언했다.
- **수정**: 서수는 "~만 확정" 같은 선택 표지에 **결합된** 것만 인정하도록 정규식을 조이고, 그 함정 문장 자체를 회귀 테스트로 박제했다(tests/test_history_focal.py). 이후 이 영역을 건드린 모든 변경은 이 테스트를 통과해야 했다.
- **가져갈 것**: 래칫과 회귀 테스트는 역할이 다르다. **래칫은 "지금 이 변경이 손해인가"를 잡는 사후 안전망이고, 테스트는 "과거에 밟은 지뢰를 다시 밟는가"를 잡는 재발 방지다.** 둘 중 하나만 있으면 구멍이 난다. 래칫만 있으면 같은 지뢰를 코드가 바뀔 때마다 다시 밟고(점수는 지켜지지만 시간을 태운다), 테스트만 있으면 새로운 유형의 퇴행을 놓친다.

### 사례 3. 검증 스크립트의 버그 — "결과가 이상하면 도구부터 의심하라"

- **오류**: 실험 토글(E5)의 효과를 확인하려고 짠 diff 스크립트가 답안 딕셔너리의 키를 잘못 읽어서(리스트/딕셔너리 혼동, 전 항목이 `None` 키로 뭉개짐) 700건 중 마지막 1건만 비교했다. 판정: "토글이 아무 효과 없음."
- **신호**: **사전 지식과의 모순.** 나는 직전에 "이 토글을 켜면 클래스 분포가 움직인다"를 다른 방법으로 확인해둔 상태였다. 분포는 움직이는데 diff가 0? 이 위화감은 하니스의 어떤 장치도 만들어주지 않았다. 순수하게 "두 관측이 서로 안 맞는다"는 사람의 감각이었다.
- **수정**: 스크립트를 고치고, 이후 모든 실험 변형은 **두 개의 독립 경로로 검증**(diff 계수 + 분포 변화)하는 걸 관행으로 삼았다. 유사 사건이 두 번 더 있었다 — 파이썬 정규식 `\w`가 한글까지 매칭해서 focal 교차검증이 전량 불일치로 보인 사건(실제로는 전량 일치였다), printf 이스케이프 문제로 기록 파일이 깨진 사건.
- **가져갈 것**: **AI가 만드는 오류의 무게중심은 하니스가 커질수록 '문제 풀이'에서 '측정 도구'로 이동한다.** 문제 풀이 오류는 점수가 알려주지만, 측정 도구 오류는 잘못된 결론을 조용히 생산한다. 결론이 이상하면 결론을 의심하기 전에 그 결론을 만든 파이프라인을 의심하는 순서를 습관화해야 한다. 이상적으로는 "중요 판정은 독립 2경로 측정"을 하니스 절차에 넣는 것.

### 사례 4. 토글 시스템의 자기 발등 — 인프라도 테스트 대상이다

- **오류**: 실험 토글이 늘면서 `SCPC_EXP=E5X`가 `"E5" in exp` 부분문자열 검사에 걸려 E5까지 같이 켜지는 버그를 심었다. 별건으로, 구버전 코드로 생성된 실험 파일을 제출 직전까지 갈 뻔한 일도 있었다(코드는 새로 고쳤는데 변형 CSV는 재생성 안 함).
- **신호**: 부분문자열 버그는 새 토글을 추가하다가 **코드 리뷰 중** 발견(운이 좋았다). stale 파일은 "매 코드 변경 후 전 변형 재생성"을 습관화한 뒤로 구조적으로 차단.
- **수정**: 토글 파싱을 쉼표 구분 토큰 집합(`exp_flags()`)으로 리팩터. 변형 생성을 항상 일괄 루프로(개별 생성 금지).
- **가져갈 것**: **하니스의 인프라 코드는 하니스가 지켜주지 않는다.** 래칫도 회귀 테스트도 하니스 본체(harness.py)만 감시했지, 실험 장치 자체는 감시 대상이 아니었다. 인프라가 어느 규모를 넘으면 인프라에도 같은 규율(테스트, 재현성 체크)을 적용해야 한다. "측정 장비의 교정"이라는 개념이 소프트웨어에도 그대로 필요하다.

### 사례 5. 캐스케이드 도달률 착시 — 추정 금지, 실측 강제

- **오류**: 분류기가 규칙 30개짜리 우선순위 폭포가 된 후반, Fable은 새 규칙의 영향 범위를 "45건"으로 추정했다. 실측 6건. 다른 규칙은 "65건" 추정, 실측 6건. 상류 규칙들이 대상을 전부 선점하고 있었는데, 모델은 새 규칙을 고립된 명제로만 추론했다.
- **신호**: **제출 전 diff 계수.** "변형 생성 → 기준판과 diff 개수·방향 계수 → 예측과 대조"를 제출 전 의무 절차로 만들어둔 게 착시를 잡았다. 이 절차가 없었으면 "45건짜리 대형 베팅"인 줄 알고 슬롯을 태웠을 거다.
- **수정**: 규칙 추가 시 영향 범위를 머리로 추정하는 것 자체를 폐기. 무조건 돌려서 센다.
- **가져갈 것**: **시스템 복잡도가 임계를 넘으면 그 시스템에 대한 언어적 추론은 (사람이든 AI든) 계통적으로 과신에 빠진다.** 유일한 해독제는 실행이다. 하니스 설계 관점에선 "예측을 기록하고 실측과 대조하는" 칸을 절차에 넣는 게 핵심이다 — 예측이 자꾸 틀리는 걸 눈으로 봐야 추정 습관이 죽는다.

### 사례 6. "신호부재"의 화석화 — 기록 시스템의 배신

- **오류**: 시도 레지스트리에 "이 축은 신호부재"로 기각 기록된 영역이 시간이 지나며 "검증된 불모지"처럼 굳었다. 실제로는 "record **타입** 수준에서 신호부재"였을 뿐인데, 기록에 분석 층위 정보가 없으니 모델은 세션마다 그 목록을 읽고 "이미 다 해봤음"으로 탐색 공간을 스스로 좁혔다.
- **신호**: **사람의 반복된 위화감.** "0.85~0.87에 사람들이 몰려 있는데 나만 카테고리 하나를 통으로 날리는 거 아냐?" — 하니스 어디에도 이 신호는 없었다. 순위 분포라는 하니스 밖 정보와, "AI가 너무 일찍 접는다"는 패턴 기억이 결합된 순수 인간 신호였다.
- **수정**: 사람이 동결을 무시하고 재공략을 지시 → 타입이 아닌 **값 조합** 수준에서 완전 판별자들이 쏟아짐(분류 잔여 12→1, 이후 형태소 수준에서 또 한 번). 이 대회 최대의 점수 점프 두 번이 전부 여기서 나왔다.
- **가져갈 것**: 기각 기록의 스키마에 반드시 **분석 층위와 재개봉 조건**을 넣어야 한다. `"결론": "신호부재"`가 아니라 `"결론": "타입 수준 신호부재", "미시도": ["값 조합", "형태소"], "재개봉 조건": "타 영역에서 값-수준 판별자 발견 시"`. 재시도 **금지**가 아니라 재공략 **조건 명시**가 올바른 설계다. 이건 이번 하니스의 가장 뼈아픈 설계 결함이었다.

### 사례 7. 전역 설정과의 상호작용 — 기계가 검증하는 규칙만 살아남는다

이건 오류라기보다 흥미로운 관찰인데, 프로젝트 하니스(Layer 1~4)와 내 로컬 루트의 전역 설정(Layer 0, `~/.claude/rules/common/`)이 유기적으로 어떻게 작용했는지 지켜본 결과다.

- **현상**: 전역 규칙들의 운명이 극명하게 갈렸다.
  - `hooks.md`의 git 커밋 컨벤션(conventional commits, 타입/스코프/제목 형식) — **수십 번의 커밋에서 거의 완벽하게 관철됐다.**
  - `coding-style.md`의 "불변성 필수, 뮤테이션 금지" — 하니스 코드는 세션 메모리를 딕셔너리 뮤테이션으로 관리한다. **조용히 무시됐다.** (그리고 그게 맞았다 — 대회 코드에서 함수형 순수성은 비용만 있었다.)
  - `testing.md`의 "커버리지 80%, 유닛/통합/E2E 3종" — 실제로는 회귀 잠금 테스트 4개. **아무도 그 괴리를 언급조차 안 했다.**
  - `performance.md`의 모델 선택 가이드 — 이번 프로젝트와 무관한 구세대 정보인데 매 세션 로드된다. **순수 컨텍스트 세금.**
- **신호**: 충돌이 시끄럽게 나지 않았다는 것 자체가 신호다. 전역 하니스는 프로젝트 하니스와 부딪히면 **소리 없이 진다.**
- **해석**: 갈린 기준이 뭘까. 커밋 컨벤션은 형식이 기계적으로 명확하고 매 커밋마다 즉시 적용 가능하다. 불변성·커버리지는 "판단"이 필요하고 프로젝트 맥락과 협상해야 한다. 즉 **전역 레벨에서 살아남는 규칙은 검증이 기계적인 규칙뿐이다.** 판단형 규칙은 전역에 둬봤자 장식이고, 프로젝트 레벨에서 맥락과 함께 다시 써야만 작동한다.
- **가져갈 것**: 전역 설정은 얇게 — 기계적으로 체크 가능한 컨벤션만. 판단형 원칙("좋은 코드란...")은 프로젝트 CLAUDE.md로 내려서 그 프로젝트의 언어로 다시 써라. 그리고 전역 파일도 유통기한 점검이 필요하다. 안 그러면 매 세션 낡은 정보로 컨텍스트만 태운다.

## 3. 부검의 종합 — 신호는 어디서 왔나

일곱 사례의 "발견 신호"만 모아보면 패턴이 보인다.

| 사례 | 신호의 출처 | 하니스가 잡았나 |
|---|---|---|
| 1. dev 암기 붕괴 | 외부 채점 (리더보드) | ❌ 내부 지표는 침묵 |
| 2. 서수 버그 | **래칫** (CV 하락) | ✅ 기계적으로 잡음 |
| 3. 측정 도구 버그 | 사람의 위화감 (관측 모순) | ❌ |
| 4. 토글 오발화 | 코드 리뷰 (반쯤 운) | △ |
| 5. 도달률 착시 | **제출 전 diff 계수 절차** | ✅ 절차가 잡음 |
| 6. 신호부재 화석화 | 사람의 직감 (외부 정보 결합) | ❌ 오히려 하니스가 원인 |
| 7. 전역 규칙 증발 | 사후 관찰 | ❌ 소리 없이 진행 |

하니스가 잡은 건 **자기가 측정하도록 설계된 종류의 오류뿐**이다(2, 5). 가장 비싼 오류들(1, 3, 6)은 전부 하니스 바깥의 신호 — 외부 채점, 사람의 위화감, 순위 분포 — 로 잡혔다. 이게 역U 곡선의 정체다: 하니스를 고도화한다는 건 "측정하도록 설계된 오류"의 커버리지를 넓히는 일인데, 그 과정에서 컨텍스트와 주의를 잠식해 **바깥 신호에 대한 감도를 떨어뜨린다.**

## 4. 더 쌓는다면 — L5의 설계도

그래도 다음 프로젝트에서 하니스를 더 고도화하고 싶다면, 이번 부검이 가리키는 방향은 명확하다. 포인트는 "더 많은 절차"가 아니라 **부검에서 하니스가 못 잡은 것들을 잡게 만드는 것**이다.

1. **시도 레지스트리 v2 — 층위·조건 스키마.**
   `{지문, 분석층위, 시도한 방법, 미시도 층위, 재개봉 조건, 기록일}`. 그리고 주기적으로 "미시도 층위가 남은 기각 항목"을 자동으로 다시 백로그에 밀어넣는 **재개봉 큐**. 화석화(사례 6)의 구조적 해법.

2. **제출 전 게이트의 완전 자동화.**
   지금은 절차(체크리스트)였던 것 — 발화율 계수, 기준판 diff, 스키마 검증, MD5 재현성 — 을 훅(hook)이나 스크립트 하나로 묶어 "이 게이트를 통과 못 하면 제출 파일이 생성되지 않는" 수준까지. 사례 4·5를 사람 주의력에서 기계로 이관.

3. **측정 도구의 이중화 원칙.**
   점수·diff 같은 중요 판정은 독립 구현 2개로 측정하고 불일치 시 정지. 사례 3의 해법. 비용이 들지만, 잘못된 결론 하나가 제출 슬롯 하나보다 비싸다는 걸 이번에 배웠다.

4. **예측 대장(prediction ledger).**
   모든 변경에 대해 "예상 효과"를 먼저 기록하고 실측과 자동 대조. 예측 오차의 추세가 보이면(사례 5의 10배 착시 같은) 모델의 추정 신뢰도를 정량적으로 디스카운트할 수 있다. AI의 낙관 편향을 데이터로 교정하는 장치.

5. **역할 분리 — 탐색기와 집행자.**
   가장 급진적인 안. 하니스 전체(동결 목록, 백로그, run sheet)를 로드하는 **집행 에이전트**와, 기준 문서(L1)만 들고 백지에서 데이터를 파는 **탐색 에이전트**를 분리해서 주기적으로 교차시킨다. 이번 대회에서 사람이 하던 "동결 무시하고 다시 파봐"를 구조로 만드는 것. 탐색기는 화석을 모르는 채 파야 한다 — 그게 탐색기의 가치니까.

6. **반란 예산.**
   자원(제출 슬롯, 시간)의 고정 비율을 "하니스의 현재 결론에 반하는 실험"에 강제 배정. 이번에 역베팅 프로브들(E6, E7, C1)이 전부 "현재 답이 옳다"는 확증으로 끝났지만, 그 확증 자체가 값이었다. 확증도 정보다.

## 5. 더 깎는다면 — 자유도를 위한 다이어트 설계도

반대 방향. 만약 다음 프로젝트가 탐색 비중이 큰 문제라면(초기 연구, 구조를 모르는 데이터), 이번 하니스에서 뭘 깎아야 AI의 자유도가 살아나나. 이번 관찰 기준으로 **깎아도 되는 것과 절대 깎으면 안 되는 것**이 명확히 갈린다.

**깎아야 할 것 (자유도를 먹는 성분):**

1. **run sheet의 결정 트리.** "Δ가 X면 다음은 Y" 식의 사전 분기는 집행자화의 주범이었다. 원칙 세 줄("큰 모집단부터", "슬롯=단일가설", "최종선택=최고점")만 남기고 트리는 폐기. 분기 판단은 결과를 보고 그때 하면 된다 — 그게 지능을 쓰는 자리다.
2. **세션 시작 시 로드되는 동결 목록.** attempts는 유지하되 **조회형으로 강등**한다. "새 가설을 세운 뒤, 실행 직전에 중복 여부만 조회"하는 것과 "세션 시작부터 불모지 지도를 머리에 얹고 시작"하는 건 탐색 심리가 완전히 다르다. 전자는 중복만 막고, 후자는 상상력을 막는다.
3. **백로그의 우선순위와 지시문.** 사실(발견, 미해결 질문)만 남기고 "다음엔 이걸 해라"는 매 세션 백지에서 다시 도출하게 한다. 어제의 우선순위는 어제의 정보로 만든 것이다.
4. **step 파일의 절차 상세.** 루프의 리듬(측정→개선→기록)만 남기고 각 단계의 how-to 상세는 삭제. 이번에 보니 모델은 절차가 자세할수록 절차 준수에 지능을 쓰고, 성길수록 문제에 쓴다.
5. **Layer 0의 판단형 규칙 전부.** (사례 7) 어차피 조용히 증발하면서 컨텍스트만 태운다.

**절대 깎으면 안 되는 최소핵 (자유도와 무관하게 손해만 막는 성분):**

- **래칫** — 자유로운 탐색이 안전한 이유 그 자체. 어떤 미친 시도든 최고 수위를 못 넘으면 자동 기각되니까, 역설적으로 래칫이 단단할수록 탐색은 과감해질 수 있다.
- **단일변수 원칙** — 실험의 해석 가능성. 이게 없으면 자유는 노이즈가 된다.
- **append-only LOG** — 기억. 단, 결론이 아니라 사실 위주로.
- **회귀 테스트** — 밟은 지뢰의 박제.

정리하면 이렇다: **모델의 손해를 막는 장치(래칫, 테스트, 단일변수)는 자유도와 상충하지 않는다. 자유도를 먹는 건 모델의 판단을 대체하려는 장치(결정 트리, 우선순위 지시, 동결 지도)다.** 안전장치는 남기고 조종장치를 깎아라. 이게 이번 대회에서 얻은 다이어트의 제1원칙이다.

## 6. 맺음 — 하니스란 과거를 미래에 강제하는 장치다

대회용 하니스(파이썬 프로그램)의 한계는 dev 120문제가 주는 정보의 상한이었다. 본 적 없는 조합 앞에서, 아무리 정교한 규칙 폭포도 결국 과거 데이터의 화석이었다.

AI용 하니스(프로세스)의 한계도 정확히 같았다. 과거 iteration들이 주는 결론의 상한. 본 적 없는 접근이 필요한 순간, 아무리 정교한 레지스트리와 run sheet도 결국 과거 결론의 화석이었다.

**하니스란 과거를 미래에 강제하는 장치다.** 과거가 미래와 닮은 구간에서는 강력하고, 닮지 않은 구간에서는 정확히 그만큼 해롭다. 그래서 좋은 하니스 설계의 최종 문제는 절차의 정교함이 아니라 **"우리의 과거가 어디까지 유효한가"를 판정하는 감각**이고, 부검 기록이 보여주듯 그 감각의 신호는 대부분 하니스 바깥에서 왔다. 외부 채점에서, 순위 분포에서, "이 결과 이상한데"라는 위화감에서.

하니스는 그 신호를 만들지 못한다. 다만 좋은 하니스는 그 신호가 왔을 때 **빠르고 안전하게 방향을 틀 수 있는 상태**를 유지해준다 — 래칫이 있으니 과감하게 틀 수 있고, 로그가 있으니 틀면서도 잃지 않는다. 하니스의 역할은 조종이 아니라 현가장치다. 다음 하니스는 그렇게 만들 거다.
